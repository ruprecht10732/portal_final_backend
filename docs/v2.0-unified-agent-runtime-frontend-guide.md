# v2.0 Unified Agent Runtime — Frontend Team Integration Guide

## TL;DR

**No changes required to frontend code or HTTP API calls.** The refactoring is entirely internal to the backend. All existing endpoints, request/response formats, and public APIs remain unchanged.

---

## What Changed (Internal Only)

### Before (v1.x)
- 8 eagerly-constructed agent structs in Module (`gatekeeper`, `estimator`, `dispatcher`, `auditor`, `quoteGenerator`, etc.)
- 5 separate scheduler task types (`GatekeeperRunPayload`, `EstimatorRunPayload`, etc.)
- 4 separate scheduler interfaces (`GatekeeperScheduler`, `EstimatorScheduler`, `DispatcherScheduler`, `AuditorScheduler`)
- Each agent built at module init time, consuming memory even when idle

### After (v2.0)
- 1 unified `Runtime` that dynamically builds agents on demand
- 1 unified task payload (`AgentTaskPayload`) with workspace routing
- 1 unified scheduler interface (`AgentTaskScheduler`)
- Agents constructed lazily per-request, only when a task is processed

## What the Scheduler Worker Sees (New)

### New Unified Task Processing

The scheduler worker now calls a single unified method on the Module:

```go
func (m *Module) ProcessAgentTask(ctx context.Context, payload scheduler.AgentTaskPayload) error
```

This replaces the old separate handler methods the worker used to call:
- ~~`ProcessGatekeeperRun`~~ (still exists but worker doesn't call it directly)
- ~~`ProcessEstimatorRun`~~ (still exists but worker doesn't call it directly)
- ~~`ProcessDispatcherRun`~~ (still exists but worker doesn't call it directly)
- etc.

The worker routes all agent tasks through `ProcessAgentTask`, which internally dispatches to the correct workspace based on `payload.Workspace`.

### AgentTaskPayload Schema

```go
type AgentTaskPayload struct {
    Workspace     string `json:"workspace"`               // "gatekeeper", "calculator", "matchmaker", "auditor"
    Mode          string `json:"mode,omitempty"`          // "estimator" or "quote-generator" (for calculator)
    LeadID        string `json:"leadId"`
    LeadServiceID string `json:"leadServiceId"`
    TenantID      string `json:"tenantId"`
    Force         bool   `json:"force,omitempty"`
    AppointmentID string `json:"appointmentId,omitempty"` // for auditor visit-report
    Fingerprint   string `json:"fingerprint,omitempty"`   // semantic dedup
}
```

## Deployment Considerations

### Queue Drain Required ⚠️

**Before deploying v2.0, you MUST drain the Redis task queue.**

Old tasks in the queue use deprecated payload formats (`GatekeeperRunPayload`, `EstimatorRunPayload`, etc.) that are no longer supported. Tasks enqueued before deployment will fail if processed by the v2.0 worker.

**How to drain:**
1. Stop the scheduler worker
2. Wait for all pending tasks to complete (monitor Redis/Asynq dashboard)
3. Deploy v2.0
4. Start the scheduler worker

### Environment Variables (Unchanged)

All environment variables remain the same. No new configuration is required.

### Database Migrations (None)

No database migrations are required. The refactoring is purely in-memory and affects only the Go code structure.

## FAQ

**Q: Do I need to update any frontend API calls?**  
A: No. All HTTP endpoints and request/response formats are unchanged.

**Q: Do I need to update the scheduler deployment?**  
A: Only to ensure the queue is drained before deploying. No configuration changes.

**Q: Can I still call `ProcessGatekeeperRun` directly?**  
A: Yes. The public method still exists and works identically. However, the scheduler worker now routes through `ProcessAgentTask` internally.

**Q: What happened to the old scheduler interfaces?**  
A: `GatekeeperScheduler`, `EstimatorScheduler`, `DispatcherScheduler`, and `AuditorScheduler` were removed. Use `AgentTaskScheduler` with `EnqueueAgentTask` instead.

**Q: Are the markdown workspaces affected?**  
A: No. The `agents/gatekeeper/`, `agents/calculator/`, `agents/matchmaker/`, and `agents/auditor/` directories are unchanged.
